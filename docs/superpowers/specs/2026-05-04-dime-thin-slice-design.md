## Framework Correction (load-bearing for the implementation)

**The PRD currently says "DIME-Plus" with 4 modules** (Sovereignty / War & Arcana / Prosperity / Environment & Perception). **The actual framework** per user's stated vision is **DIME+FIL+MCG** — 10 attributes split into three roles:

| Code | Name | Role | Source in thin slice |
|---|---|---|---|
| D | Diplomacy | Primary (GM slider 1-10) | User input |
| C | Culture | Primary (GM slider 1-10) | User input |
| M | Military | Primary (GM slider 1-10) | User input |
| E | Economic | Primary (GM slider 1-10) | User input |
| I | Information | Primary (GM slider 1-10) | User input |
| I² | Intelligence | Primary (GM slider 1-10) | User input |
| L | Law Enforcement | Derived (computed from Government + Religion) | Read-only display |
| F | Finance | Derived (computed from E) | Read-only display |
| M* | Magic | World-granted pool (computed from Magic Level + Culture + civ_tier tap formula) | **Stubbed** in thin slice — returns 0; no pool boost on M* |
| G | Geography | World-granted pool (from Territorial Audit) | **Stubbed** in thin slice — returns audit's gBaseline only; no chokepoint/hub bonuses |

So the GM directly sets 6 sliders + 4 dropdowns (Government / Religion / CivTier / Species-dominant). L and F derive automatically. M* and G are world-granted pools that boost M_effective / E_effective / M*_effective via tap formulas (spec §8.4) — both stubbed in thin slice (effective values equal primary values).

The 4 modules per the kickoff spec are **interview groupings** of these 10 attributes, not the framework itself. The 10 attribute heads + Head of Country = 11 named NPCs (which directly addresses Schwif-T's "5-10 key figures" content ask). Magic + Geography drive a 3-section weather forecast (per day / per season / per climate zone) which addresses the weather ask. **NPC roster and weather forecast are Phase 2 work, NOT in thin slice.**

**Action:** PRD requires a follow-up edit to capture the 10-attribute structure. Out of scope for this thin slice's code, but in scope as a docs-only edit before implementation begins.

## Approaches Considered

### Approach A — Rip-the-bandaid (selected)

- Polygon stored as GeoJSON in `NationCreated` event payload.
- Interview state in React component state — no draft persistence (refresh loses work).
- Cascading rules as pure function (5 rules from spec §10).
- Factbook as TS template fragments rendered to JSX.
- Skip: IndexedDB drafts, GM overrides, audit cache.
- Reuse: existing WorldQuery + scrubber + MapLibre.

**Effort:** 1-2 weeks human / ~30-60 min CC.
**Pros:** Schwif-T touches it fastest. Lessons learned drive v1 architecture.
**Cons:** Rebuild draft persistence + audit cache when the loop matures.

### Approach B — Architect for v1 — match the spec

- Cascading rules per spec §10 (id/when/apply/explanation, pluggable).
- Interview state in IndexedDB with `NationDraftStarted`/`NationFinalized` events.
- Pantheon dictionary as separate per-world store.
- Factbook template fragments keyed by facet level per §12.

**Effort:** 3-4 weeks human / ~2-3 hours CC.
**Pros:** v1 work is incremental from v0. Closer to canonical spec.
**Cons:** Same shape as the sphere-substrate decision — more upfront infrastructure before the user touches anything. Risk of repeating the off-rails pattern.

### Approach C — LLM-narrated factbook

- Same as A, but factbook prose generated by LLM consuming structured nation state.
- Cascading rules + derived facets stay deterministic; only the prose layer goes through the LLM.

**Effort:** 1-2 weeks human / ~1-2 hours CC.
**Pros:** Scales to full Intelligence Briefing format trivially. Previews the v0.1 LLM stack.
**Cons:** Introduces an LLM dependency PRD explicitly defers to feature #3. Adds cost-per-nation. Non-deterministic until temp=0 pinned. Premature.

## Recommended Approach

**Approach A — Rip-the-bandaid.** Selected.

Rationale: matches the lesson learned from the sphere-substrate session. The user explicitly chose to stop fighting the process and ship the user-facing loop. Approach B is the same fork that produced the off-rails detour; choosing A this time is the conscious-not-accidental version of "ship before you architect."

These supersede the "Approaches Considered" choices and the Open Questions above where they conflict.

Derived facets (L, F) computed at read time. Factbook prose rendered on read from templates. No prose pre-rendering into the payload.

- **SQL migration:** ~~~`0005_nation_event_kind.sql`~~~ **NOT NEEDED.** Verified during /writing-plans on 2026-05-05: the existing `events.kind` column in `supabase/migrations/0001_initial.sql:44` is just `text not null` with NO CHECK constraint. App-level TypeScript discriminated union is the only kind-validation. The eng review locked this migration based on an incorrect assumption.

## Appendix A — Stub specifications (engineering-normative)

### A.1 Elevation-distribution → suggestion lookup (audit stub)

**Why elevation, not biomes:** the MAURO substrate has `heightmap: Uint16Array` (16-bit elevation 0..65535 ≈ -10000m to +10000m mapped per the existing prep-tiles convention) and `mask: Uint8Array` (1=land, 0=ocean). It does NOT have biome data. Synthesizing biomes from elevation alone (without latitude, hydrology, climate) would produce misleading biome labels. Using elevation distribution directly is honest signal.

The audit returns:
```ts
interface AuditOutput {
  areaKm2: number;
  elevationDistribution: {
    deepWater: number;       // % of polygon cells where mask=0 AND elevation < threshold_deep
    shallowWater: number;    // % where mask=0 AND elevation >= threshold_deep
    lowland: number;         // % where mask=1 AND elevation in [0, 500m]
    midland: number;         // % where mask=1 AND elevation in [500m, 1500m]
    highland: number;        // % where mask=1 AND elevation > 1500m
  };
  suggestions: SliderSuggestion[];
}
```

Thresholds (in meters above sea level, after Uint16 → meters mapping per tile):
- **deep water:** elevation < -200m below sea level (mask=0)
- **shallow water:** elevation in [-200, 0) (mask=0) — coastal shelf
- **lowland:** elevation in [0, 500] (mask=1)
- **midland:** elevation in [500, 1500] (mask=1)
- **highland:** elevation > 1500 (mask=1)

Suggestion lookup table — based on the dominant land-elevation band (≥40% of *land cells*) and significant water presence (≥20% of *all cells*):

| Pattern | Suggested sliders | Prose fragment for tooltip |
|---|---|---|
| ≥40% lowland AND ≥20% water | E=5, D=6 | "Easy water access and arable lowlands favor maritime trade." |
| ≥40% lowland (no significant water) | E=5, M=5 | "Open lowlands favor cavalry and farming both." |
| ≥40% midland | E=4, M=5, D=4 | "Mixed terrain — defensible enough, productive enough, but no easy edge." |
| ≥40% highland | M=6, E=3, L=6 | "Defensible terrain, hard living, strong rule needed." |
| ≥60% water (mostly ocean) | (block submission — see Edge Cases) | — |
| Mixed (no band ≥40%) | E=4, M=5 | "Diverse terrain, no defining geographic edge." |
