## Problem Statement

MAURO has a thorough kickoff bundle (BRD, PRD, PERSONAS, ROADMAP, TECH_STACK, ARCHITECTURE_PRINCIPLES, CARRY_FORWARD) but no code. The user needs to pick the first feature: small enough to ship in 2-3 weeks, structured to prove the wedge against Inkarnate / Wonderdraft / Azgaar. The user's stated bias is the "alive world" promise (event-sourced state, nation creation, real-Earth crops) over polish/UX. The session's job is to pick the slice and surface the tradeoffs honestly.

## Demand Evidence

User has 3+ named GMs (not themselves) lined up for the beta — concrete, specific people who match the long-campaign 5E persona. Of the three pillars (timeline scrubbing / CIA-factbook nations / real-Earth geography), the pillar that lit those GMs up was **"build worlds from real Earth."** This is hard demand evidence: not a survey, not a waitlist signup, but a pitch-test where one specific message converted three specific people.

Note: this evidence contradicts the user's *stated* prioritization (alive-world). The session resolved the tension in favor of the demand signal — see Premise 3.

## Status Quo

The primary persona currently juggles three tools: Notion (or Obsidian) wiki for nation lore, Inkarnate (or Wonderdraft) for the map, hand-drawn timeline for "as of when" tracking. None of these talk to each other. None of them produce real-Earth-derived geography. None of them version the world. The cost: re-keying the same nation across three artifacts; "what did the world look like in Session 12" is a memory-and-spreadsheet exercise; the map looks generic-fantasy instead of feeling like a specific place.

## Target User & Narrowest Wedge

**The human:** A GM running a 12+ month campaign with named factions and political tension, currently maintaining a Notion wiki + Inkarnate map + hand-drawn timeline. They lit up at "build worlds from real Earth."

**The wedge:** A world-creation flow that takes 2-3 real-Earth heightmap tiles (cropped from NASA SRTM / GEBCO / ETOPO), composites them into a single fantasy continent rendered in MapLibre, and demonstrates one working event-sourced mutation of that world (e.g., coastline shift, lake fill, volcanic eruption). The world has a current-date display and a scrubber between two timestamps showing the pre/post-event state.

The narrowness is deliberate: ONE event type, TWO timestamps, ZERO nation-creation UI. The composer is the hero; the event is the moat-glimpse that signals "this is the alive-world thing, even if you can't see it yet."

## Premises (agreed)

1. Stated bias and demand evidence diverged. Session resolved in favor of demand evidence.
2. The first feature has one job: get the next GM to try MAURO. The moat reveals itself in week 4 of beta, not in the first demo.
3. "Build worlds from real Earth" was undervalued by the user, correctly weighted by the GMs. Real terrain has fractal correlations and drainage patterns subliminally legible to humans — procgen looks "off" even when polished.
4. A 2-3 week feature cannot ship event-sourced state alone. The minimum aliveness demo requires geography + an event + a scrubber. Any pure-aliveness slice is bigger than the budget.
5. Empty monorepo overhead compounds scope risk. ~10 days of actual feature time, not 14-21.

**User refinement to Premise 1:** "Agree mostly — but I want the moat visible too." The first feature must include a working aliveness glimpse (a single event-sourced mutation), not just a pretty map. This shaped the recommended approach.

## Approaches Considered

### Approach A: First Map, First Event (RECOMMENDED — selected)

Earth tile composer (crop / rotate / composite ≤3 tiles) → MapLibre render → world has current-date display → ONE pre-defined event type that mutates geography via the event log → scrubber slider between two timestamps. XL effort, Med risk. Hits hook + moat-glimpse.

### Approach B: Earth Composer Only

Just the composer + render. Ship in ~2 weeks, low risk, no aliveness visible. Rejected: violates the user's "moat visible too" refinement; ships a tool that competes with Inkarnate on Inkarnate's terms.

### Approach C (lateral): World Bible Genesis

Curated Earth tiles (no composer) + nation polygon drawing + event-sourced creation-history scrubber. Maximum substrate validation, but defers the demo hook the GMs reacted to. Rejected: trades away the demand signal; aliveness here is creation-log, which sets wrong precedent for what aliveness means in MAURO.

## Recommended Approach

**Approach A: "First Map, First Event" — scope-cut variant.**

The original Approach A scoped a multi-tile composer + Edge Function tile pipeline + MapLibre custom heightmap source. Adversarial review (see Reviewer Concerns below) showed this was 1.5-2x over budget AND contained a stack mismatch (`sharp` is Node-native, Supabase Edge Functions are Deno). The variant below preserves the hook + moat-glimpse intent while cutting the load-bearing risks.

**Key cuts vs. original A:**
- No live tile composer. User picks ONE tile from a curated list of 3-5 pre-rendered choices.
- No Edge Function tile crop. Tiles are pre-cropped offline by a one-time `scripts/prep-tiles.ts` Node job and committed/uploaded as static raster + heightmap pairs.
- No MapLibre custom heightmap source. Use a plain MapLibre `raster` source over a pre-rendered hill-shaded PNG. The demo event swaps the raster URL based on the scrubber timestamp.

The composer ships in v0.1, after the substrate is validated end-to-end.

6. **World creation flow (~1 day).** "New World" page: pick one tile from the 3, set name + magic-level, click create. Persists a `WorldCreatedEvent` and a row in `worlds` (the row is a denormalized cache of the latest `WorldCreatedEvent`, not source of truth — see Data Model).

7. **World detail page + MapLibre raster render (~1.5 days).** MapLibre with a single `raster` source. The source URL is `/api/render/{substrateHash}.png` resolved from `WorldQuery.getWorldAsOf()`. Mask-as-source-of-truth (Architecture Principle #6) is enforced because the rendered hill-shade is composed from the mask layer at prep time, not from elevation thresholding.

8. **Demo event + scrubber + write-time render (~3 days; was 2.5).** `GeographyMutationEvent` payload: `{ kind: 'volcanic_uplift', polygonId: string, elevationDelta: number, atDate: ISO8601 }`. Pre-bound polygon hardcoded per tile (one valid polygon per region, picked at tile-prep time in QGIS or hand-coded coords with a comment in `prep-tiles.ts`). **Render happens at write-time, not read-time:** when `WorldCreatedEvent` is inserted, a Vercel API route computes the T0 substrate hash and re-runs the hillshade pipeline (the same one used in prep-tiles, exported from `packages/geo`) on the post-replay heightmap, then uploads the T0 hill-shade PNG to Supabase Storage. When `GeographyMutationEvent` is inserted, the same route renders/uploads the T1 PNG. Both PNGs exist before the user ever drags the scrubber. There are only 2 PNGs per world in v0 — the read-side render-cache and concurrency-race problems collapse entirely. Scrubber: continuous slider between T0 (world creation) and T1 (+50y after demo event). Slider position → maps to one of two known URLs. The +0.5 day vs round 2 covers the custom hillshade integration in the Vercel route.

### Architecture exercises (validates principles before they harden)

- **Substrate-first (#1):** the rendered hill-shade PNG shown in MapLibre is a render of the substrate (`WorldCreatedEvent` + event log replayed against the chosen tile's heightmap + mask), not the substrate itself.
- **WorldQuery API (#2):** the scrubber CANNOT reach into the events table directly — must go through `WorldQuery.getWorldAsOf()`. This is the API's first user.
- **Time-versioned (#3):** the demo event is an append to the event log. There is no "update the world's elevation" mutation. The "after" state is computed by replay.
- **Determinism (#4):** scoped to **substrate hash, not pixel hash**. Same world + same event sequence → identical SHA256 of the heightmap bytes. The MapLibre client render is GPU-driver-dependent and is explicitly out of the determinism contract.
- **State/prose separation (#5):** not exercised in v0 because no LLM ships. Recorded here so the principle isn't forgotten when LLM lands in feature #2 or #3.
- **Mask-source-of-truth (#6):** the rendered hill-shade is composed at prep time from the source tile's is-land mask layer, NOT from elevation thresholding. The volcanic_uplift event modifies elevation only — it does not touch the mask.

### WorldQuery Contract

**Module layout:**
- `packages/sim/src/types.ts` — `Workspace`, `World`, `WorldEvent`, `EventRow`, `WorldSnapshot`.
- `packages/sim/src/events/applyEvent.ts` — pure reducer (lives with event types, not with query).
- `packages/sim/src/query/WorldQuery.ts` — class that does the replay-via-SupabaseClient work.
- `packages/sim/src/query/factories.ts` — two factories that pick the right Supabase client for the context (see below).

**WorldQuery construction — TWO factories, not a generic constructor.** Supabase SSR has two distinct clients (server-with-cookies for RLS-protected user-scoped reads in app routes; service-role for RLS-bypassing reads in the render webhook). One generic constructor silently leaks across contexts. Force the call site to declare intent:

```ts

import { createServerClient } from '@supabase/ssr';
import { createClient } from '@supabase/supabase-js';

export function worldQueryForUser(req: Request): WorldQuery {
  // Reads cookies from req, builds the user-scoped server client. RLS-respected reads.
  return new WorldQuery(createServerClient(/* ... */));
}

export function worldQueryForServiceRole(): WorldQuery {
  // Builds the service-role client using SUPABASE_SERVICE_ROLE_KEY (server-only env var).
  // ONLY for trusted server contexts: the render webhook, the workspace-init trigger, etc.
  // Bypasses RLS. NEVER call from a code path that runs with user input.
  return new WorldQuery(createClient(process.env.NEXT_PUBLIC_SUPABASE_URL!, process.env.SUPABASE_SERVICE_ROLE_KEY!));
```

### Hillshade Implementation

`sharp` is a 2D image library with no built-in hillshade — it knows pixels, not elevation. Hillshade requires elevation-aware operations. The spec uses `sharp` for I/O (read 16-bit grayscale heightmap PNG → raw `Uint16Array`; write 8-bit RGBA hillshade PNG) and a custom JS module for the actual shading.

**Module:** `packages/geo/src/hillshade.ts` (new — currently `tile-source.ts` is the only source file there). Exports:

```ts
type HillshadeParams = {
  azimuthDeg: number;       // sun direction in degrees, 0=N, 90=E. Default 315 (NW), the cartographic convention.
  altitudeDeg: number;      // sun height above horizon. Default 45.
  zFactor: number;          // vertical exaggeration. Default 1.0; 2-3 for low-relief regions.
  cellSizeMeters: number;   // ground distance per pixel. Comes from tile resolution (SRTM 1-arcsec ≈ 30m at equator).
};

export function computeHillshade(
  heightmap: Uint16Array,
  width: number,
  height: number,
  mask: Uint8Array,           // is-land mask; ocean pixels rendered as flat blue
  params: HillshadeParams,
): Uint8Array;                // RGBA, length = width * height * 4
```

**Algorithm (Horn's method, the standard):**

1. For each pixel, compute slope and aspect from the 3×3 neighborhood (Sobel-like convolution against the heightmap, scaled by `cellSizeMeters` and `zFactor`).
2. Compute the dot product of the surface normal with the sun vector (from azimuth + altitude).
3. Clamp to [0, 1], multiply by 255 → grayscale shade value.
4. Composite: land pixels get the shade value as RGB (with alpha 255); ocean pixels (per mask) get a fixed blue (e.g., `#1a3a5c`) instead of being shaded.
5. Edge pixels (the 1-pixel border) use replicate-padding for the convolution to avoid black borders.

**Why Horn over Lambertian-only:** Horn's method handles the slope-aspect-azimuth math correctly. Lambertian alone would compute light intensity but not direction. Horn is what `gdaldem hillshade` implements; matching it makes future migration to GDAL trivial if perf demands it.

**Performance budget:** A 2048×2048 tile (~4M pixels) at ~100ns per pixel for the convolution + dot product = ~400ms on a Vercel function CPU. Well inside the 10-second function timeout. No streaming or chunking needed for v0.

**Used by:**
- `scripts/prep-tiles.ts` — to produce the T0 hillshade PNG when no event has been replayed (the world's "as created" state, against the source heightmap).
- `app/api/events/render-on-write/route.ts` — to re-shade after `applyEvent` mutates the heightmap.

Both call paths import the SAME `computeHillshade()` so `(prep, render)` outputs are pixel-identical for an unchanged heightmap. This is asserted as a test (see Test Plan).

**Per-tile metadata (lighting + demo polygon):** `HillshadeParams` defaults are not one-size-fits-all — Patagonian fjords (~3000m relief) look great at zFactor=1; central Asian massif (~7000m) needs zFactor≈0.5; low-relief regions need 2-3. Each prep'd tile ships a sibling `tile.json` colocated with the heightmap+mask PNGs in Supabase Storage:

```ts
// uploaded by scripts/prep-tiles.ts; read by render-on-write and by the world detail page.
type TileMetadata = {
  slug:
    | 'earth-patagonia' | 'earth-norway' | 'earth-pamirs'
    | 'mars-tharsis' | 'moon-imbrium';
  body: 'earth' | 'mars' | 'moon';     // for UI grouping + attribution wording
  sourceRegion: { name: string; lat: number; lon: number; widthDeg: number; heightDeg: number };
  cellSizeMeters: number;       // derived from source resolution
  hillshadeParams: HillshadeParams;
  demoPolygon: {                // pre-picked at prep time, in pixel coords
    polygonId: string;
    pixels: Array<[number, number]>;   // closed polygon in tile-local pixel space
    description: string;               // for UI label, e.g. "Coastal lowland east of Skyhold"
  };
  // Provenance + license — required, not optional. Surfaced in UI per body
  // ("data via NASA SRTM" for Earth; "data via NASA MOLA" for Mars; etc.).
  source: {
    dataset: 'SRTM' | 'GEBCO' | 'ETOPO' | 'MOLA' | 'LOLA' | 'SLDEM2015';
    datasetVersion: string;            // e.g. "SRTMGL1.003", "MOLA-MEGDR-2003-128px"
    downloadUrl: string;               // canonical source for reproducibility
    fileChecksum: string;              // SHA256 of the downloaded source file (exact-pin per AP #4)
    license: 'public-domain';
    attribution: string;               // e.g. "Courtesy of the U.S. Geological Survey" or "NASA / MOLA Science Team"
  };
};
```

**Why provenance is required, not optional:** SRTM, GEBCO, and ETOPO are all public-domain BUT NASA/USGS/NOAA request attribution per use ("Courtesy of the U.S. Geological Survey" for SRTM, etc.). MAURO's UI must display the source attribution next to any rendered tile (e.g., a footer line on the world detail page). The `fileChecksum` field also satisfies the byte-affecting-dependencies pin policy from `docs/ARCHITECTURE_PRINCIPLES.md` §8 — input data is byte-affecting; checksumming the downloaded source means tile regeneration is reproducible.

The render route reads `tile.json` via Storage (cached in-memory per cold start) before calling `computeHillshade(heightmap, w, h, mask, tile.hillshadeParams)`. Adding a 4th tile means: drop a new `{slug}/tile.json` + heightmap + mask in Storage, add the slug to the `World['tileSlug']` union, ship a migration that updates the CHECK constraint. No code changes to the math module.

**Render lifecycle (write-time, synchronous in the writer route):**

Outside voice (round 3) flagged that the trigger+webhook design solved a multi-producer problem v0 doesn't have. v0 has exactly two write paths (`POST /api/worlds` and `POST /api/worlds/[id]/events`), both Vercel routes you control. The simpler, more transactional design:

- Each writer route does, atomically: (a) call a Supabase Postgres function (RPC) `create_world_with_event(...)` or `add_event(...)` that inserts the world+event in one DB transaction and returns the new event row. (b) Compute the post-replay substrate hash. (c) Run the hillshade pipeline. (d) Upload `{substrateHash}.png` to Supabase Storage `tiles-rendered/` with `upsert: true`. (e) Return success to the client only after Storage upload completes.
- If steps (c)-(d) fail, the route returns a 5xx; the client retries (idempotent, since substrateHash is deterministic and Storage `upsert: true` is safe). The DB row stays — there is no `render_status` column, no `render_error`, no failure state machine. A failed render means "the next request that reads this world will trigger the render synchronously again" (see fallback below).
- **Read-side fallback (defensive):** if `WorldQuery.getWorldAsOf()` computes a `substrateHash` whose PNG is missing in Storage (the rare case where a writer route returned 5xx after committing the event), the read path renders synchronously inline before returning the snapshot. Costs 400-1000ms on the read but recovers the broken-image failure mode without infrastructure.
- In v0 every world has exactly 2 PNGs (T0 + T1). No triggers, no webhooks, no DB-Webhook config, no HMAC verification, no `render_status` column, no failure-UI state.
- **Renderer runs in Vercel API routes (Node), not Supabase Edge Functions (Deno) — `sharp` is Node-native.**

**Why this matters for the data model:** the `render_status`, `render_error`, `rendered_at` columns and the `events_render_status_pending` index are removed from the schema sketch (see Data Model below — corrected). The `auth.users` workspace-creation trigger remains (different concern; that one IS a multi-producer case).

## Test Plan (must-haves before ship)

Implementation must write tests alongside feature code, not defer to a follow-up. Coverage diagram in `/plan-eng-review` identified ~50 testable paths; the list below names the ones that ship as part of v0. Vitest for unit/integration; Playwright for E2E.

**Determinism (`packages/sim/src/rng/`):**
1. **splitmix64 reference vectors** (already passing — `splitmix64.test.ts` exists). Verify against the canonical Sebastiano Vigna reference output.
2. **xoshiro256\*\* reference vectors:** seed `[1,2,3,4]` produces the published reference output for the first 16 calls to `next()`.
3. **xoshiro256\*\* state-init from splitmix64:** `new Xoshiro256(splitmix64(seed))` is byte-identical across runs.

**Reducer (`packages/sim/src/events/applyEvent.ts`):**
4. **Volcanic uplift mean-elevation:** `applyEvent` with `elevationDelta=400` increases mean elevation INSIDE polygon by exactly 400 units (within rounding); outside-polygon pixels unchanged.
5. **Mask immutability:** the input `mask` Uint8Array reference and bytes are unchanged after `applyEvent` returns (assert via byte equality + length).
6. **Polygon clipping correctness:** for a polygon covering exactly N pixels, exactly N pixels are mutated; boundary pixels follow inclusive-edge rule.
7. **Elevation 16-bit upper clamp:** `elevationDelta` that would push a pixel above `UINT16_MAX` clamps to `UINT16_MAX`, never overflows.
8. **Elevation 16-bit lower clamp:** negative `elevationDelta` clamps at 0, never underflows.
9. **Unknown event kind:** `applyEvent({ kind: 'NotARealKind' as any })` throws a typed error (defensive contract for future event types).

**WorldQuery (`packages/sim/src/query/`):**
10. **Replay determinism:** given `(world, [WorldCreated, GeographyMutation])`, `getWorldAsOf(t2)` produces stable substrate hashes across 100 runs.
11. **As-of correctness across event boundary:** for events at `t1 < t2`, `getWorldAsOf(t1)` excludes the t2 event; `getWorldAsOf(t2)` includes both.
12. **Initial state:** `getWorldAsOf(asOfDate < world.createdAt)` either throws `WorldNotYetCreated` or returns the source-tile-only snapshot (decide and assert).
13. **Not-found:** `getWorldAsOf(unknown_world_id, ...)` throws `WorldNotFound` (typed).
14. **render_status passthrough:** when latest replayed event has `render_status='pending'`, `WorldSnapshot.renderState === 'pending'`. Same for `'rendered'` and `'failed'` (with `renderError` populated).
15. **Factory: worldQueryForServiceRole missing env var:** throws on construction if `SUPABASE_SERVICE_ROLE_KEY` is unset (catches deploy misconfiguration early).

**Hillshade (`packages/geo/src/hillshade.ts`):**
16. **Flat plane:** uniform-elevation heightmap → uniform shade output (within ±1 byte for floating-point edge effects).
17. **Sun overhead:** `altitudeDeg=90` → no shadows, all land pixels equally lit.
18. **East-facing slope, sun in west:** synthetic gradient heightmap with sun at azimuth=270 → west-facing pixels brighter, east-facing pixels darker.
19. **Ocean pixels per mask:** mask=0 pixels render as the fixed ocean color (`#1a3a5c`), regardless of underlying heightmap value.
20. **Output buffer length:** `output.length === width * height * 4` (RGBA).
21. **Edge replication:** 1-pixel border has no black edge artifact (assert top-left, top-right, bottom-left, bottom-right pixels are within ±1 byte of their inward neighbor).
22. **Prep ↔ render parity:** `computeHillshade` called from `prep-tiles.ts` and from `render-on-write/route.ts` produces byte-identical output for an unchanged heightmap (key invariant for substrate-first principle).

**Database (`supabase/migrations/`):**
23. **RLS isolation:** user A's anon-key read of `worlds` returns 0 rows when only user B's worlds exist; same for `events` and `workspaces`. Service-role read returns all rows.
24. **Workspace auto-creation trigger:** inserting a row into `auth.users` (via Supabase admin API in test fixture) creates exactly one matching `workspaces` row in the same transaction.
