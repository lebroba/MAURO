# Design: Sphere Substrate Primitives

Generated by /brainstorming on 2026-05-01
Branch: main
Repo: MAURO (E:\projects\MAURO)
Status: PROPOSED — pending user review

## Problem Statement

MAURO will reach planetary-scale rendering and simulation in v1+ (true-globe worlds, plate tectonics, multi-tile composition, climate strata). Architecture Principle #10 — added 2026-05-01 — establishes that world data is spherical and 2D rasters are renders of that substrate. Five sub-rules follow: Euler-rotation tectonics from day one, `(lon, lat)` as the canonical coord system, horizontal seam continuity, cosine-latitude weighting, and polar zones as render-distortion zones by policy.

The MVP per the PRD ships single-tile worlds (Patagonia, Norway, Pamirs, Tharsis, Imbrium) where most of these rules don't yet bite. But by v1, every consumer that touches geometry will need sphere-aware primitives. Without those primitives in place from day one, v1 work either gets blocked or accumulates "we'll fix the projection later" debt.

This spec covers building the substrate primitives that unblock v1 — without yet shipping any v1 features. The library is forward investment: pure functions, no consumers in MVP, validated against a planet-scale test harness so its API survives first contact with v1 consumers.

## Scope

**In scope:**
- A pure-function library at `packages/sim/src/sphere/` covering coords, geodesy, area, noise, distribution, and WGS84 ellipsoid math.
- A planet-scale validation harness at `packages/sim/src/sphere/__tests__/characteristic.test.ts`.
- Two new pinned dependencies: `simplex-noise@4.0.x`, `geographiclib-geodesic@2.0.x`.
- A small audit pass on existing `packages/sim/` files to flag any load-bearing flat-Earth assumptions and document MVP-safe shortcuts.

**Out of scope:**
- Plate tectonics simulation itself (v1 — the rotation primitives are built; the simulation isn't).
- Multi-tile composition / GlobalField abstraction (v1 — substrate enables it; consumer code is separate).
- Climate strata painting (v1+ — substrate exposes `latitudeBand`; the climate algorithm is separate).
- Any change to `SubstrateState`, `WorldQuery`, `applyEvent`, or `TileLoader` other than audit JSDoc comments.
- The existing `apps/web/src/app/poc/stitch/` page — sandbox, untouched per Q3 decision.

## Premises (agreed during brainstorm)

1. **Scope: B from Q1.** Substrate primitives + lightweight retrofit pass. Approach A (primitives only) leaves existing code as a future hazard; C (use-case-led) under-builds; D (pull plate tectonics into MVP) breaks the PRD without cause.
2. **World endpoint: C from Q2.** Composed-tile world is the MVP shape; true-globe is the v1 endpoint. Substrate primitives are designed for the v1 endpoint from day one but no v1 consumer is built now.
3. **Stitch-poc treatment: A from Q3.** `/poc/stitch` stays as a public sandbox demo. Production substrate is built fresh, separately. No code coupling to the demo page.
4. **Climate strata is the canonical use case.** Mentioned by the user as the load-bearing reason sphere-aware substrate matters. Substrate primitives expose what climate (v1+) will need: `latitudeBand`, area-weighted accumulators, cosine-weighted distributions. The climate algorithm is out of scope for this spec.
5. **Approach: 2.** Bottom-up library + planet-scale validation harness. Approach 1 risks API drift; Approach 3 over-fits to one not-yet-existent feature.
6. **Hybrid coordinate frame policy** (decided after WGS84 question). Distance/area/ECEF use WGS84 ellipsoid; rotation/slerp/noise/plate-tectonics use unit sphere. Standard practice in geo software — captures the real-Earth-units concerns without forcing ellipsoidal analogs of operations that don't have them.
7. **Noise library: simplex-noise v4** (Wagner). Picked over Perlin because lattice anisotropy artifacts manifest as faint spurious banding when 3D noise is sampled on a sphere; Simplex's isotropic lattice eliminates this. Library has clean determinism ergonomics (custom RNG hookup) and is small and pure-JS.

## Architecture & Layering

The sphere library is a leaf module inside `packages/sim`, sibling to `rng/`, `events/`, `query/`. It owns no state, performs no I/O, and depends on nothing else in `sim`. Other modules (existing or v1) import *from* it; it imports nothing back.

Three coordinate systems coexist intentionally:

| System | Type | Used for |
|---|---|---|
| **Geodetic `(lon, lat)`** | `LonLat = { lonDeg, latDeg }` — `[-180, 180)`, `[-90, 90]` | Substrate API surface. All public function inputs/outputs |
| **Cartesian unit sphere** | `Cartesian3 = { x, y, z }` on the unit sphere | Internal math: rotations, distances, noise sampling, slerp |
| **WGS84 ECEF Cartesian** | `ECEF = { x, y, z }` in meters from Earth center, geodetic-correct | Real-world 3D position; export, GIS interop |
| **Tile-local pixel** | `TilePixel = { px, py }` paired with existing `TileMetadata` | Storage and raster I/O only — `TileLoader`, render path, source SRTM data |

Conversions are explicit and the only places these systems talk:

- `lonLatToCartesian(LonLat) → Cartesian3` (unit sphere) and `cartesianToLonLat(Cartesian3) → LonLat`
- `lonLatToECEF(LonLat, heightMeters?) → ECEF` (WGS84) and `ecefToLonLat(ECEF) → LonLat` (Bowring's iterative formula)
- `lonLatToTilePixel(LonLat, TileMetadata) → TilePixel | null` (returns null if outside the tile's `sourceRegion`)
- `tilePixelToLonLat(TilePixel, TileMetadata) → LonLat`

Inside any algorithm, pick one frame and stay in it. API boundaries are LonLat. Rotations and distances are unit-sphere Cartesian. Real-world position is ECEF. Raster reads are tile-pixel. The raster is a translation layer at the storage edge, never the working representation.

**Numeric type:** `number` (float64) throughout. No `Float32Array` in this library — float32 accumulates error fast under repeated rotations (rule 10a). The few places float32 is needed (GPU, PNG packing) are at render boundaries.

**No new package.** Lives at `packages/sim/src/sphere/` to follow existing convention. Imported as `import { ... } from './sphere'` from inside `@mauro/sim`. No top-level re-export from the package's `index.ts` until v1 has a real consumer (avoids freezing the API too early).

## Components

Seven files, one job each. All pure functions. No I/O. No state.

### `wgs84.ts` — WGS84 ellipsoid constants

```ts
export const WGS84 = {
  A_METERS: 6378137.0,           // equatorial radius
  F: 1 / 298.257223563,          // flattening
  // Derived (precomputed at module load):
  B_METERS: number,              // polar radius = A * (1 - F)
  E2: number,                    // first eccentricity squared = 2F - F²
  E_PRIME2: number,              // second eccentricity squared = E² / (1 - E²)
} as const
```

No functions. Constants only. Used by `coords.ts` (ECEF conversions), `geodesy.ts` (geodesic distance), and `area.ts` (ellipsoidal cell area).

### `coords.ts` — types and lossless conversions

```ts
export interface LonLat   { lonDeg: number; latDeg: number }
export interface Cartesian3 { x: number; y: number; z: number }   // unit sphere frame
export interface ECEF       { x: number; y: number; z: number }   // WGS84 frame, meters
export interface TilePixel  { px: number; py: number }

export function lonLatToCartesian(p: LonLat): Cartesian3
export function cartesianToLonLat(p: Cartesian3): LonLat
export function lonLatToECEF(p: LonLat, heightMeters?: number): ECEF
export function ecefToLonLat(p: ECEF): LonLat                      // Bowring's iterative formula
export function lonLatToTilePixel(p: LonLat, meta: TileMetadata): TilePixel | null
export function tilePixelToLonLat(px: TilePixel, meta: TileMetadata): LonLat
export function normalizeLon(deg: number): number                  // wraps to [-180, 180)
export function clampLat(deg: number): number                      // clamps to [-90, 90]
```

`Cartesian3` and `ECEF` share a `{ x, y, z }` shape but document different frames. No branded types — every signature documents the frame in a JSDoc comment. The bug class we care about is unit confusion (degrees vs radians, frame vs frame), which we solve with field naming (`lonDeg`, `angleRad`, `radiusMeters`).

`normalizeLon` exists because float drift across rotations produces `180.0000001`-shaped values; one canonical wrap point prevents seam bugs.

### `geodesy.ts` — great-circle and geodesic math

```ts
export function greatCircleDistanceMeters(a: LonLat, b: LonLat, radius?: number): number
export function geodesicDistanceMeters(a: LonLat, b: LonLat): number   // WGS84 via Karney
export function slerp(a: Cartesian3, b: Cartesian3, t: number): Cartesian3
export function rotateAxisAngle(p: Cartesian3, axis: Cartesian3, angleRad: number): Cartesian3
export function eulerPoleRotation(p: LonLat, pole: LonLat, angleRad: number): LonLat
```

`greatCircleDistanceMeters` uses sphere math (Haversine), fast, used for "is this within X of that" checks where ellipsoid precision is irrelevant. Default `radius` is the WGS84 mean radius (`6371008.8` m) — the standard "spherical Earth" approximation. `geodesicDistanceMeters` uses WGS84 via the `geographiclib-geodesic` library (Karney's algorithm — antipode-safe, ~0.3% more accurate than sphere over Earth-scale distances).

`eulerPoleRotation` is a thin LonLat wrapper around `lonLatToCartesian → rotateAxisAngle → cartesianToLonLat`. This is rule 10a's foundation. Plate tectonics in v1 will call this in a loop per cell per timestep; the geometry is here from day one.

**Slerp antipode handling.** Standard slerp divides by zero at antipodal inputs. Implementation:

```ts
function slerp(a, b, t) {
  const cosΩ = dot(a, b)
  if (cosΩ > 1 - EPSILON_SAME)
    return normalize(lerp(a, b, t))                        // nearly identical
  if (cosΩ < -1 + EPSILON_ANTIPODAL)
    return rotateAxisAngle(a, perpendicularFallback(a), t * Math.PI)   // antipodal
  // standard slerp formula
}
```

`perpendicularFallback(a)` uses a fixed convention: cross with `(1,0,0)` unless `a` is parallel to it, in which case cross with `(0,1,0)`. Deterministic, reproducible, perpendicular to `a`.

### `area.ts` — cell geometry and latitude classification

```ts
export function cellAreaSterad(latDeg: number, dLatDeg: number, dLonDeg: number): number
export function cellAreaSqMeters(latDeg: number, dLatDeg: number, dLonDeg: number, radius?: number): number
export function cellAreaSqMetersWGS84(latDeg: number, dLatDeg: number, dLonDeg: number): number
export type LatitudeBand = 'tropical' | 'subtropical' | 'temperate' | 'subpolar' | 'polar'
export function latitudeBand(latDeg: number): LatitudeBand
export function isPolarZone(latDeg: number): boolean    // |lat| >= 80, per rule 10e
```

Spherical cell area uses Archimedes' hat-box theorem: `dA = R² · |sin(lat₁) − sin(lat₂)| · dLon`. Default `radius` for `cellAreaSqMeters` is the WGS84 mean radius (`6371008.8` m). Ellipsoidal cell area in `cellAreaSqMetersWGS84` integrates the WGS84 surface element using the meridional and prime-vertical radii of curvature — no `radius` parameter; the WGS84 ellipsoid constants are used directly.

`latitudeBand` thresholds use standard climatology: tropical [0°, 23.5°], subtropical [23.5°, 35°], temperate [35°, 55°], subpolar [55°, 66.5°], polar [66.5°, 90°]. Uses `|lat|` so northern and southern hemispheres map to the same bands. The polar-zone-by-policy classifier (rule 10e) is independent of the climatological "polar" band — `isPolarZone` is the rendering-policy classifier (`|lat| >= 80°`).

### `noise.ts` — sphere-native procedural noise

```ts
export interface SphereNoiseParams {
  seed: bigint
  octaves: number
  frequency: number
  lacunarity: number
  persistence: number
}

export function sampleSphereNoise(p: LonLat | Cartesian3, params: SphereNoiseParams): number
```

OpenSimplex/Simplex (`simplex-noise@4.0.x`) evaluated at the Cartesian unit-sphere point. Because the input space is 3D and continuous, dateline + pole singularities don't exist — they're artifacts of 2D-pixel-space noise that we never use.

Seed integration: derive the noise's internal permutation table from a `xoshiro256ss` instance seeded with the user's `bigint`. The library accepts a custom RNG (a `() => number` returning [0, 1)), so we wrap our existing RNG and pass it in.

If `p` is `LonLat`, internally converted to `Cartesian3` once; if already Cartesian, passed through. Output range is `[-1, 1]`.

### `distribution.ts` — sphere-aware random placement

```ts
export function uniformOnSphere(rng: Xoshiro256ss): LonLat
export function cosineWeightedPoisson(
  rng: Xoshiro256ss,
  count: number,
  minSeparationRad: number,
): LonLat[]
export function areaWeightedAccumulate<T>(
  cells: Iterable<{ latDeg: number; value: T }>,
  combine: (acc: T, value: T, weight: number) => T,
  initial: T,
): T
```

`uniformOnSphere` uses `(2π·u, acos(2v−1))` with `u, v ∈ [0, 1]` from the RNG. Correct uniform-on-sphere; uniform-in-lat-lon would cluster at poles (rule 10d violation).

`cosineWeightedPoisson` does dart-throwing with great-circle min-separation: draw a candidate via `uniformOnSphere`, accept if no existing point is within `minSeparationRad` great-circle distance. Caps at a fail-budget to prevent infinite loops on over-dense parameters.

`areaWeightedAccumulate` is the generic reducer for any "sum or average over a global field" use case. The combine function receives the area weight (in steradians) for each cell. Used by climate (v1+), terrain analysis, anything that needs cell-area awareness.

### `index.ts` — public surface

Re-exports the public functions from the above files. Internal helpers (e.g., `perpendicularFallback`, math primitives like vector ops) stay un-exported.

No top-level re-export from `@mauro/sim`'s `index.ts` until v1 has a consumer. Avoids freezing the API too early.

## Data Flow

Three representative call chains showing how primitives compose:

**A. Place 50 magic nodes uniformly across a planet.**
```ts
const rng = xoshiro256ss(splitmix64(masterSeed, 'magic-nodes'))
const nodes: LonLat[] = cosineWeightedPoisson(rng, 50, /*minSeparationRad*/ 0.05)
```
Single primitive call. Cosine weighting is hidden inside; caller never sees the distortion math. RNG is passed in, never owned by the library.

**B. Sample sub-tile biome noise at a queried point.**
```ts
const cart = lonLatToCartesian({ lonDeg: -73.5, latDeg: -41.2 })
const value = sampleSphereNoise(cart, { seed, octaves: 4, frequency: 8, lacunarity: 2, persistence: 0.5 })
```
LonLat in (API surface), Cartesian internally for sphere-correct math, scalar out. No wrap or pole concerns surface to the caller — they're impossible by construction.

**C. v1 plate tectonics step (illustrative — not built in this spec).**
```ts
for (const cell of plate.cells) {
  cell.position = eulerPoleRotation(cell.position, plate.eulerPole, plate.angularVelocityRad * dt)
}
```
Whole-plate rotation is one primitive call per cell. When v1 builds the tectonics simulation, the only new code is the plate model and integration loop. The geometry is already done.

**Threading through the existing substrate.** `SubstrateState`, `WorldQuery`, and `applyEvent` are unchanged by this spec. The current rectangular `Uint16Array` heightmap + mask is tile-local pixel data and stays that way. Sphere primitives operate alongside it. When v1 needs a "global field" abstraction (planet-wide heightmap composed from multiple tiles), it'll be a new type built on top of these primitives, designed then.

**Determinism flow.** Every primitive that reads randomness takes an `Xoshiro256ss` instance from the caller. None own RNG state. Stage-seed derivation stays in the caller. The library is trivially testable with a mock RNG; consumers control their own determinism boundaries.

**No-side-effects invariant.** Every public function is pure: same inputs → same outputs, no I/O, no logging, no caching, no mutation of inputs. Validation is at boundaries (`normalizeLon`/`clampLat` after operations that could produce out-of-range values), not at every call.

## Determinism & Types

**Contract.** Same inputs → byte-identical outputs across runs, same V8 version, same architecture. Cross-arch verification (x86 vs ARM) remains a v1 target per Architecture Principle #9; MVP ships on Vercel x86.

**Pin policy.**
- `simplex-noise@4.0.x` exact (whatever the latest 4.0 patch is at adoption).
- `geographiclib-geodesic@2.0.x` exact.
- Both pinned via `--save-exact` in `packages/sim/package.json` adjacent to existing pinned deps.
- Bumping either requires a vector-test refresh and a determinism-contract entry — same workflow as the existing RNG pins per Architecture Principle #8.

**Numeric types.**
- Everything `number` (float64 implicit).
- No `Float32Array` anywhere in this library.
- `bigint` only for seed values, matching existing `xoshiro256ss` signature.

**No branded types.** Interfaces over classes. Unit confusion is prevented by field naming (`lonDeg`, `angleRad`, `dLatDeg`, `radiusMeters`). Frame confusion (`Cartesian3` vs `ECEF`) is prevented by JSDoc on every signature.

**Vector tests as determinism net.** Every public primitive gets a golden-vector test: frozen reference output for known seed + known input, asserted byte-identical. A breaking change to any primitive — even a "harmless refactor" — fails a vector test and forces an explicit acknowledgement in the PR. Same pattern the existing `xoshiro256.test.ts` uses.

**Math.sin/cos caveat.** V8 uses fdlibm for trig — portable across platforms, but not formally bit-exact-guaranteed across CPU vendors. In practice it's been consistent on Vercel's runtimes. Documented assumption; cross-arch verification (Principle #9) will catch any drift.

## Validation Harness

Lives at `packages/sim/src/sphere/__tests__/characteristic.test.ts`. Marked slow — runs in pre-merge CI, not on every save. The harness exercises primitives end-to-end in a planet-shaped way, catching composition bugs that unit tests miss.

Six test families:

**1. Round-trip & invariants.**
- `lonLatToCartesian → cartesianToLonLat` at 1000 random points: max error < 1e-12 degrees
- `lonLatToECEF → ecefToLonLat` at 1000 random points + heights from −500 m to +8848 m: max error < 1e-9 degrees, height < 1e-3 m
- `slerp(a, b, 0)` ≈ `a`, `slerp(a, b, 1)` ≈ `b`
- `rotateAxisAngle(p, axis, 2π)` ≈ `p` (full rotation, error < 1e-10)
- `eulerPoleRotation` over 100 steps of θ vs one step of 100θ: end positions match within 1e-9 degrees (rotational composition associativity)

**2. Antipode handling.**
- 100 antipodal pairs: `slerp(a, antipode, 0.5)` returns deterministic perpendicular midpoint. Same input → same output across runs. Output is unit-length and perpendicular to both input vectors.
- `slerp` near-antipodal (cosΩ in `[-1 + ε, -1 + 1e-3]`) transitions smoothly into the standard formula: no discontinuity at the threshold.

**3. Known-value tests.**
- Great-circle NYC (-74°, 40.7°) → London (0°, 51.5°) ≈ 5,570 km within 5 km (sphere)
- Geodesic NYC → London ≈ 5,585 km within 1 km (WGS84) — the 0.3% delta is the ellipsoid
- Equator quarter-circle (0,0) → (90,0): π/2 · R = 10,018,754 m within 1 m
- `cellAreaSqMeters` summed over a 1° resolution global grid ≈ 4πR² within 0.01%
- `cellAreaSqMetersWGS84` summed over a 1° grid ≈ ellipsoid surface area (≈ 510,065,621 km²) within 0.01%

**4. Continuity at the seams.**
- `sampleSphereNoise(lon=179.99, lat=0)` and `sampleSphereNoise(lon=-179.99, lat=0)`: |Δ| < 0.01
- 10 random `(lonDeg, 89.999)` points: noise values within tight band (continuous near pole because Cartesian positions are nearly identical)
- Repeat for south pole

**5. Distribution statistics.**
- 10,000 `uniformOnSphere` points: bucket by latitude band, each bucket count ÷ total ≈ band's share of total surface area, within ±2σ binomial variance
- 10,000 points: bucket by 5° lon × 5° lat cells, average points per cell weighted by `cellAreaSterad` is constant across cells
- `cosineWeightedPoisson(rng, n=200, minSep=0.1)`: all pairwise great-circle distances ≥ 0.1 rad

**6. Composition with real tile catalog.**
- For each tile (Patagonia, Norway, Pamirs, Tharsis, Imbrium): low-res sample of `(px, py) → (lon, lat) → Cartesian → LonLat → (px, py)` round-trip. Max pixel error < 1px.
- Synthetic two-tile-overlap test: same `(lon, lat)` reached via each tile's `tilePixelToLonLat` produces byte-identical noise samples.

**Output artifacts.** Pass/fail per family logged. Plus `characteristic-stats.json` artifact with actual values (distance histograms, distribution counts) — detects statistical drift over time even when tests still pass. Becomes a regression net for any future v1 work.

## Architecture Principle Linkage

This spec implements Architecture Principle #10 (added to `docs/ARCHITECTURE_PRINCIPLES.md` 2026-05-01). Sub-rule mapping:

| Sub-rule | Substrate primitive that enables it |
|---|---|
| 10a — Plate tectonics as Euler rotations on a sphere | `geodesy.ts:eulerPoleRotation` |
| 10b — Coordinate system is `(lon, lat)` not `(x, y)` | `coords.ts:LonLat`, all public APIs |
| 10c — Horizontal seam continuity is a substrate invariant | `noise.ts:sampleSphereNoise` (3D-Cartesian sampling) |
| 10d — Cosine-latitude weighting on density and area | `distribution.ts:uniformOnSphere`, `cosineWeightedPoisson`, `areaWeightedAccumulate`; `area.ts:cellArea*` |
| 10e — Polar zones are render-distortion zones by policy | `area.ts:isPolarZone` |
