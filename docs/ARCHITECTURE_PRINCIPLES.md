# MAURO — Architecture Principles

Load-bearing rules. Violations need explicit justification, ideally captured as a follow-up principle.

1. **Substrate-first, render-second.** The data structure (cell graph + event log) is the product. The heightmap raster is one render of the substrate, not the canonical form.
2. **WorldQuery API is the foundation.** All reads go through it. Pipeline stages do not reach sideways into each other's data — only through the API. (Diagnosis from project_aria 2026-04-25 pivot: most regressions came from cross-stage data coupling.)
3. **Time-versioned state by default.** Mutable in-place updates forbidden at the persistence layer. Every change is an event with a timestamp.
4. **Determinism contract for procgen and rules-engine paths.** Same inputs (master seed, world params, event sequence) produce byte-identical outputs across runs and architectures. RNG is splitmix64-derived stage seeds → xoshiro256\*\* stage RNG.
5. **State and prose are separate.** Rules engine produces state deltas deterministically. LLM produces narrative prose. State is load-bearing; prose is regeneratable. Storing the prose is a cache, not a source of truth.
6. **Mask is source-of-truth for is-land. Elevation only provides relief.** (Carry-forward lesson — when elevation sign was used to gate land/ocean, swiss-cheese coastlines resulted.)
7. **GPL clean-room.** Concepts from devlogs / academic papers OK. Never read source from GPL-licensed procgen tools (SotE, SongsOfGPL, Gleba, etc.). Documented in `CONTRIBUTING.md`.
8. **Exact-pin policy for byte-affecting dependencies.** Any library that affects output bytes (RNG, noise, geo) is pinned exactly. Policy is documented in the manifest itself so contributors don't "modernize" it.
9. **Cross-architecture CI** (target: v1, not MVP). Verify byte-identical output on x86 + ARM via output-hash assertions.
10. **World data is spherical; 2D is a render.** The world is stored as a sphere parameterized by `(lon, lat)`. Equirectangular raster is one render of that sphere, not the canonical form. Switching to globe presentation must be a projection swap, not a data migration. Five rules follow from this and are load-bearing — violating any of them creates retrofit cost ranging from "render fix-up" to "rewrite the geological substrate":
    - **a. Plate tectonics as Euler rotations on a sphere from day one.** Plates rotate around an axis through the planet's center, not translate as 2D rigid bodies. 2D translation is geometrically wrong globally and no projection trick fixes it. This is the only sub-rule that genuinely cannot be retrofitted; everything else is a render concern.
    - **b. Coordinate system is `(lon, lat)`, not `(x, y)`.** Internal storage uses spherical coordinates throughout. Equirectangular `(x, y)` exists only inside render code. WorldQuery API surfaces lon/lat.
    - **c. Horizontal seam continuity is a substrate invariant.** Any noise (Perlin, Simplex, Worley) wraps at lon = ±180. Perlin/Simplex: sample on a 2D circle in 4D noise space. Worley: mirror seed points across the seam. Every procgen stage either wraps or fails a unit test — no seam-broken stage ships.
    - **d. Cosine-latitude weighting on density and area-dependent computations.** A 1°×1° cell at lat 60° has half the equatorial area, zero at the poles. Uniform 2D scatter clusters at the poles when wrapped. Every "place N things uniformly" call weights probability by `cos(lat)`. Erosion iteration counts and rainfall accumulation also weight by cell area.
    - **e. Polar zones (top/bottom ~10° lat) are render-distortion zones by policy.** No important named features in polar zones. Heightmap tends to a near-constant value approaching the pole. Climate is uniform-cold. This is acceptance of equirectangular pole degeneracy, not a bug to fix.

    **Hex grid tradeoff (decide later, not now):** Regular 2D hex grids do not wrap to spheres. Goldberg hexes (icosphere dual) are sphere-native but incompatible with 2D hex math. Working resolution that preserves 5E TTRPG ergonomics: world map uses rectangular `(lon, lat)` cells; regional/tactical maps are local hex projections (transverse Mercator centered on the region) entered via an explicit "zoom in" projection switch. Do not try to make one hex grid serve both scales.
